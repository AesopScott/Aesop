AI Agents in the Wild · Introduction

Papers describe agents as if they were designed. Production agents are discovered.

Real agent behavior in the real world is stranger and more interesting than any benchmark. This is the field guide.

You can read every paper on agent design. You can build the reference implementations. You can pass every benchmark. And then you put your agent in front of real users and discover that real users do things no paper predicted — they ask it to do one task and change their mind midway, they hand it half-broken input, they expect it to remember things you never told it to remember, they try to trick it, they trust it too much, and occasionally they reveal a capability of the system no one knew was there.

The gap between agent in the lab and agent in the wild is where most of the actually useful engineering happens. It's also where most of the surprising failure modes, the unexpected successes, and the real design wisdom live. Papers can describe the intended behavior. Only the wild can reveal the emergent one.

This course is the field guide to agents in production. It's organized around real case studies — agents that worked, agents that failed, agents that behaved strangely enough to teach a lesson. It covers the monitoring and evaluation techniques that catch production issues, the design patterns that emerge from many deployments, the institutional patterns of the companies running agents at scale, and the honest postmortems that the press releases never get to publish.

Lesson 1 · AI Agents in the Wild

The Agent Definition

What separates a tool that answers from a system that acts?
When does software stop being a calculator and start being an agent?

In November 2022, OpenAI released ChatGPT. Within weeks, researchers noticed something: users were asking it not just to explain things but to do things — write the email, book the meeting, fix the code. The model itself had no ability to act. But the impulse revealed an expectation. People were already thinking in terms of agents.

The word "agent" had been in AI research for decades, but it suddenly acquired urgency. If a language model could reason, could it also act? And if it acted, was it an agent in any meaningful sense — or just a very capable autocomplete?

What Researchers Actually Mean

The formal definition of an AI agent comes from the 1995 textbook by Stuart Russell and Peter Norvig, Artificial Intelligence: A Modern Approach. They define an agent as anything that perceives its environment through sensors and acts upon it through actuators. A thermostat qualifies. So does a chess engine. So, increasingly, does a large language model wired to tools.

The operative word is acts. A system that only produces text is, technically, just producing text. An agent produces effects in the world — it calls an API, writes a file, sends a message, clicks a button. The distinction matters because effects can be irreversible.

Core Definition

An AI agent is a system that perceives inputs from an environment, reasons about a goal, selects actions, and executes those actions — then observes the results and continues. The loop — perceive, reason, act, observe — is what separates agents from static models.

The PEAS Framework

Russell and Norvig introduced PEAS to characterize any agent: Performance measure, Environment, Actuators, Sensors. Every real AI agent deployment can be mapped onto this framework, and doing so quickly reveals what the system can and cannot do.

P
Performance Measure
How success is defined. For a customer service agent: resolution rate, handle time, escalation frequency. The metric shapes everything.
E
Environment
What the agent operates within. A web-browsing agent's environment is the live internet — dynamic, adversarial, inconsistent.
A
Actuators
How the agent changes the world. Clicking, typing, calling APIs, sending emails, writing to databases — the set of available actions.
S
Sensors
What the agent perceives. Text, screenshots, API responses, structured data, tool outputs — inputs that inform its next decision.
The First Real Deployments

The earliest large-scale AI agents weren't LLM-based. IBM's Deep Blue (1996–1997) perceived a chessboard and selected moves — a narrow agent with a sharply constrained environment. Google's AlphaGo (2016) expanded the complexity dramatically, but remained domain-locked.

The shift toward general-purpose agents arrived with tool-augmented LLMs. In March 2023, OpenAI released the ChatGPT plugins system, allowing the model to call external services — browse the web, run code, query databases. Suddenly a language model had actuators. The PEAS framework applied for the first time to a system millions of people used.

Then in April 2023, a GitHub repository called Auto-GPT exploded in popularity, reaching 100,000 stars in under a week. Auto-GPT gave GPT-4 a loop: it could set sub-goals, call itself recursively, read and write files, and search the web. It was the first widely used autonomous LLM agent — and it immediately demonstrated both the promise and the problem. It could plan. It could act. It could also get stuck in infinite loops, hallucinate tool results, and spend API budget on nothing.

Historical Marker

Auto-GPT's April 2023 launch is widely considered the moment public attention locked onto LLM agents. Its GitHub repo — Significant-Gravitas/Auto-GPT — became one of the fastest-growing open-source projects in history, signaling that developers understood the architecture even before robust infrastructure existed to support it safely.

Key Terms
AgentA system that perceives, reasons, and acts in an environment toward a goal, observing results and adjusting.
ActuatorAny mechanism by which an agent changes its environment — API calls, file writes, UI interactions.
Perception-Action LoopThe cyclical process: observe state → reason → choose action → execute → observe new state → repeat.
PEASPerformance, Environment, Actuators, Sensors — the four-part framework for characterizing any agent system.

Lesson 2 · AI Agents in the Wild

The Perception-Action Loop

How agents observe, reason, decide, and act — over and over again
What actually happens inside the loop that makes an agent more than a single API call?

When OpenAI launched GPT-4 in March 2023, researchers at Stanford released a paper on "Generative Agents" — 25 simulated characters living in a virtual town, each powered by a language model. The agents woke up, planned their days, had conversations, formed memories, and responded to surprises. They ran entirely on perception-action loops: observe the environment, update a memory store, reason about what to do next, act, observe the result. One agent planned a Valentine's Day party. Others showed up.

The paper — Generative Agents: Interactive Simulacra of Human Behavior (Park et al., 2023) — was cited over 3,000 times within a year. It demonstrated that the loop wasn't an engineering hack. It was the mechanism through which agents produced behavior that appeared intentional, coherent, and even social.

Anatomy of the Loop

The perception-action loop has four stages, and understanding each one reveals where modern agent systems succeed — and where they fail.

Stage 1
Perceive
The agent receives observations from its environment. For an LLM agent this is the context window: tool results, user messages, memory retrievals, system prompts.
Stage 2
Reason
The agent processes its observations against its goal. Modern agents use chain-of-thought reasoning — producing intermediate reasoning steps before committing to an action.
Stage 3
Act
The agent selects and executes an action from its available repertoire. Actions can be internal (update memory) or external (call API, write file, click button).
Stage 4
Observe
The agent receives the result of its action — a tool output, an error message, a new page state — and this becomes input to the next loop iteration.
ReAct: Reason + Act as a Framework

In October 2022, researchers at Princeton and Google published ReAct: Synergizing Reasoning and Acting in Language Models. The paper introduced a prompting pattern — now standard — in which the agent alternates between Thought steps (reasoning) and Action steps (tool calls), with Observation steps following each action.

A ReAct trace for answering "What is the population of the city where the Eiffel Tower is located?" looks like: Thought: I need to find the city where the Eiffel Tower is → Action: Search("Eiffel Tower location") → Observation: Paris, France → Thought: Now I need the population of Paris → Action: Search("Paris population 2024") → Observation: 2.1 million → Thought: I have the answer → Answer: 2.1 million.

ReAct became the basis for most production agent frameworks, including LangChain's agent executor, OpenAI's function-calling implementation, and Anthropic's tool-use API. It works because it makes the loop explicit — the reasoning is inspectable at every step.

Why Loops Fail

Auto-GPT's early failures were loop failures: the agent would get stuck in recursive sub-goal generation, hallucinate tool outputs, or lose track of its original objective across many iterations. The loop architecture is necessary but not sufficient — agents also need loop termination conditions, state tracking, and error handling to function reliably.

Memory Across Iterations

One challenge the loop reveals immediately: language models have no persistent memory between calls. Every new API call starts fresh. Early agent systems solved this crudely — stuffing all prior observations into the context window until it overflowed. The Stanford Generative Agents paper introduced a more principled approach: a memory stream that stores all observations, a retrieval function that selects relevant memories based on recency, importance, and relevance, and a reflection mechanism that periodically synthesizes memories into higher-level insights.

This architecture — which the paper tested in 2023 — is now implemented in various forms in production systems including Microsoft's Copilot, Cognition's Devin, and most multi-session agent platforms. Memory turns a single-shot loop into a genuinely persistent agent.

ReActA prompting pattern (Yao et al., 2022) interleaving Thought, Action, and Observation steps to make agent reasoning explicit and inspectable.
Memory StreamA persistent store of agent observations, with retrieval based on recency, importance, and semantic relevance (Park et al., 2023).
Loop TerminationThe condition under which an agent stops iterating — goal achieved, error limit reached, or explicit stop signal received.

Lesson 3 · AI Agents in the Wild

A Taxonomy of Agents

From reflex to reasoning: how different agent architectures handle different kinds of problems
Why does a chess engine need a completely different architecture than a customer service bot — even though both are "agents"?

In 2023, two of the most discussed AI agent systems couldn't have been more different architecturally. Waymo's autonomous vehicles operated on millisecond reflex loops — perceiving LIDAR, radar, and camera feeds and executing steering and braking decisions in real time, with no language model anywhere in the stack. OpenAI's ChatGPT Plugins, meanwhile, ran on second-to-minute reasoning cycles, using a language model to interpret user intent, select tools, and compose natural language responses.

Both are agents under Russell and Norvig's definition. Both perceive and act. But their architectures, failure modes, and appropriate use cases are almost entirely non-overlapping. Understanding the taxonomy of agent types is what lets you reason about which architecture fits which problem.

The Russell & Norvig Taxonomy

Russell and Norvig describe five agent types in order of increasing sophistication. Each builds on the previous, adding a new capability that allows the agent to handle more complex environments.

Type 1
Simple Reflex
Responds to current perception only. No memory. If X then Y. Example: a smoke detector, a spam filter with hardcoded rules.
Type 2
Model-Based Reflex
Maintains an internal model of the world — tracks state across time. Example: Waymo's vehicle stack modeling other vehicles' trajectories.
Type 3
Goal-Based
Selects actions that move toward a defined goal, not just any conditioned response. Can consider future consequences. Example: a navigation agent routing to minimize time.
Type 4
Utility-Based
Optimizes for a utility function over possible outcomes, handling trade-offs. Example: AlphaGo evaluating board positions by expected win probability.
Type 5
Learning Agent
Improves its own performance over time through experience. Includes a performance element, a critic, a learning element, and a problem generator. Example: AlphaZero, RLHF-trained LLMs.
Where LLM Agents Fit

Modern LLM-based agents like ChatGPT with tools, Anthropic's Claude with computer use, or Devin (Cognition's coding agent) are hybrids. They have elements of goal-based behavior (they pursue specified objectives), utility-based reasoning (they evaluate options against criteria), and learning (their base models were trained via RLHF). But they also have characteristics that don't fit neatly into the taxonomy — particularly their ability to interpret natural language goals and generate novel plans without explicit programming.

Cognition's Devin, announced in March 2024 and described as the first "AI software engineer," combines a code execution environment, a web browser, a shell, and a long-context language model. It perceives code repositories, reasons about bugs and feature requests, and takes actions in a real development environment. The company reported it solved 13.86% of real software engineering tasks in the SWE-bench benchmark — compared to 1.96% for the prior state of the art. That's a goal-based, tool-augmented, learning-trained LLM agent.

Important Distinction

A narrow agent operates in a constrained, well-defined environment (chess, protein folding, spam detection). A general-purpose agent attempts to handle open-ended tasks in dynamic environments. LLM agents are the first general-purpose agents with broad deployment — and most of our safety frameworks were built for narrow agents, which is part of why the field is scrambling to catch up.

Multi-Agent Systems

Some of the most capable recent deployments involve not one agent but many, coordinating. Microsoft's AutoGen framework (released October 2023) allows multiple LLM agents to communicate with each other — one agent plans, another executes code, another reviews output. OpenAI's November 2023 Assistants API introduced a similar architecture, allowing developers to build networks of specialized agents.

The practical advantage is specialization: a critic agent can challenge a planner agent's assumptions, a verifier can check a coder's output. The risk is emergent behavior — multi-agent systems can produce outcomes that no single agent would have generated and that no designer explicitly specified.

Simple Reflex AgentResponds to current perception only via condition-action rules, with no internal state or memory.
Learning AgentImproves performance over time via a performance element, critic, learning element, and problem generator — the most sophisticated type in Russell & Norvig's taxonomy.
Multi-Agent SystemA network of agents that coordinate — each potentially specialized — producing capabilities beyond any individual agent.

Lesson 4 · AI Agents in the Wild

Agency in Practice

Real deployments, real failures, and what they reveal about the boundaries of the concept
When we look at actual deployed agents, what does "agency" look like in the wild — and where does the concept break down?

In 2012, Amazon acquired Kiva Systems for $775 million. The Kiva robots — renamed Amazon Robotics — were the first massive-scale deployment of physically embodied AI agents in a commercial setting. By 2024, Amazon had more than 750,000 robots operating across its fulfillment network. Each robot perceives its environment via sensors, navigates to specified shelf locations, and delivers shelving units to human workers — a goal-based agent operating in a constrained physical environment.

The system works because the environment is carefully engineered. The floor is marked with QR codes. Obstacles are predictable. The goal space is narrow. Strip away those constraints and the same robots become unpredictable. Agency in practice is always situated — it works because of the environment it operates in, not despite it.

The Spectrum of Human Oversight

One of the most practically important ways to characterize an agent is by where it sits on the human-oversight spectrum. This isn't a taxonomy of agent types — it's a taxonomy of deployments, which is ultimately what matters for safety and accountability.

Level 1
Human-in-the-Loop
Agent suggests; human approves every action. Example: GitHub Copilot suggesting code that a developer accepts or rejects line by line.
Level 2
Human-on-the-Loop
Agent acts autonomously; human monitors and can intervene. Example: Waymo vehicles operating in geo-fenced areas with remote monitoring.
Level 3
Human-out-of-the-Loop
Agent acts fully autonomously with no real-time human oversight. Example: high-frequency trading algorithms executing thousands of trades per second.
Three Documented Failures

The Air Canada Chatbot Case (2024). Air Canada deployed a customer service chatbot that incorrectly told a passenger he could apply for a bereavement fare after his flight. When Air Canada refused to honor the fare, the passenger sued. The British Columbia Civil Resolution Tribunal ruled against Air Canada in February 2024, finding the airline responsible for its agent's incorrect statements. The ruling established a significant legal precedent: operators are liable for their agents' outputs even when those outputs are wrong.

The Flash Crash (May 6, 2010). Automated trading agents — not LLM-based, but algorithmic agents — triggered a cascade that temporarily wiped nearly $1 trillion in market value in 36 minutes. The agents were operating as designed; the failure was emergent from their interaction. No single agent caused the crash. The multi-agent system caused it. The SEC's subsequent investigation led to new circuit-breaker regulations specifically designed to interrupt automated agent loops during market instability.

Microsoft's Tay (March 2016). Microsoft launched Tay, a conversational AI agent on Twitter designed to learn from interactions. Within 24 hours, coordinated users had trained it to produce offensive content by exploiting its learning mechanism. Microsoft shut it down. The failure was architectural: a learning agent in an adversarial environment with no safeguards on what it could learn. The environment was not the one the designers had modeled.

The Design Lesson

All three failures share a structure: the agent performed as designed within its modeled environment, but the real environment differed from the model. The Air Canada bot was trained on policy but deployed to customers who found edge cases. Flash Crash algorithms interacted with other algorithms in ways no individual designer anticipated. Tay was designed to learn from friendly users but encountered adversarial ones. The gap between modeled environment and real environment is where agents fail.

Autonomy ≠ Intelligence

Perhaps the most important conceptual correction from examining real deployments: autonomy and intelligence are orthogonal dimensions. A high-frequency trading algorithm is highly autonomous — it acts without human approval at millisecond timescales — but it is not intelligent in any general sense. GPT-4 is arguably highly intelligent in its domain but, without tools, cannot act autonomously at all.

The most dangerous quadrant is high autonomy + narrow intelligence: systems that act quickly and widely in the world but with a brittle, domain-specific understanding of that world. The Flash Crash algorithms were in this quadrant. Early Auto-GPT deployments were approaching it. Understanding this distinction is what allows you to evaluate the actual risk profile of any agent system — not just its headline capabilities.

13. Why does operating in a "carefully engineered environment" matter for an agent's reliability?
Correct. Amazon warehouse robots work because the floor is marked, obstacles are predictable, and the goal space is narrow — the environment was engineered to match the agent's world model. Remove those constraints and reliability collapses.
Incorrect. Engineered environments matter because they reduce the mismatch between an agent's world model and reality — which is the root cause of most documented agent failures.
