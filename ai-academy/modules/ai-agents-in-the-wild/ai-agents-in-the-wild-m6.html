Module 6 · Lesson 1

Orchestrator–Subagent Architecture

How one agent commands many — and why the division of labor matters.
When is it smarter to split a task across multiple specialized agents than to build one monolithic agent?

When OpenAI demonstrated its Operator agent publicly in late 2024, the most striking feature was not what it could do alone — it was how it spun up sub-processes for distinct subtasks: one thread navigating a browser, another reasoning over the extracted text, a third formatting the output. A single user prompt produced a choreographed team, not a lone actor.

What Is an Orchestrator?

An orchestrator is an agent whose primary job is to decompose a goal and delegate subtasks to specialized subagents. It does not usually perform domain work itself; it plans, routes, monitors, and synthesizes. Subagents receive scoped instructions, execute within their specialty, and return results to the orchestrator.

This mirrors the structure of a competent project manager: the PM does not write every line of code or draft every legal clause — they coordinate the people who do.

Orchestrator

Receives top-level goal. Decomposes into subtasks. Routes, monitors, and synthesizes subagent outputs. Holds the high-level plan.

Subagent

Receives a scoped instruction. Executes with tools appropriate to its specialty. Returns structured output to the orchestrator.

Tool Layer

APIs, databases, browsers, code interpreters. Subagents call tools; they rarely talk to each other directly without orchestrator mediation.

Real Architectures in Production

Google's AlphaCode 2 (December 2023) used a multi-agent pipeline where a planning agent first decomposed a competitive programming problem into sub-goals, a coding agent generated candidate solutions, and a separate critic agent filtered them. The critic was not retrained for each problem — it was a standing subagent with a fixed evaluation role. This separation allowed DeepMind to improve each component independently.

Cognition Labs' Devin (March 2024) publicly demonstrated an orchestrator that managed a planning subagent, a shell-execution subagent, and a browser subagent — each with its own context window and toolset. The orchestrator maintained a shared scratchpad that all three could read, but only the orchestrator could write final instructions.

Anthropic's internal research on multi-agent Claude (2024) documented that orchestrator–subagent splits reduced token waste on long-horizon tasks because subagents held only task-relevant context, not the entire project history.

Why It Matters

When a subagent receives a prompt from an orchestrator, it cannot verify that the orchestrator has not itself been compromised. Anthropic's Claude usage policy explicitly states that Claude should refuse unsafe instructions regardless of whether the instruction arrives from a human or another agent. Trust does not transfer automatically through the chain.

Key Terms
OrchestratorAn agent that decomposes goals and delegates subtasks to subagents, synthesizing their outputs into a coherent result.
SubagentAn agent that executes a scoped task within a multi-agent pipeline, typically reporting back to an orchestrator.
Context ScopingGiving each subagent only the information relevant to its subtask, reducing noise and token costs.
Trust Propagation RiskThe danger that malicious or erroneous instructions from one agent are blindly executed by downstream agents in the chain.
Historical Parallel

The orchestrator–subagent pattern is not new to computer science. Unix pipes (1973) chain specialized single-purpose programs. MapReduce (2004) assigns a coordinator to route data across worker processes. Modern multi-agent AI systems have rediscovered this architecture at the semantic level.

Communication Protocols Between Agents

How agents pass messages, share state, and coordinate without colliding.
What determines whether agents in a system collaborate effectively or devolve into conflicting, redundant work?

Microsoft Research's AutoGen framework, released in September 2023, tackled a concrete coordination problem: when you give two LLM-based agents a shared task, how do they avoid interrupting each other or repeating work? The team's answer was a structured conversation protocol — each agent had a defined speaking turn, a reply schema, and a termination condition. Without this, the agents entered loops, contradicted each other, and never converged.

The Communication Layer

Multi-agent communication is not merely passing strings between processes. It requires agreement on message format (what fields exist), turn-taking (who speaks when), shared memory access (what is readable by whom), and termination signals (how agents know a subtask is complete).

The three dominant patterns in deployed systems are: direct messaging (point-to-point between specific agents), blackboard architecture (agents read/write to a shared structured workspace), and event-driven messaging (agents subscribe to events and react when relevant signals fire).

Direct Messaging

Agent A calls Agent B with a structured message. Simple, traceable. Tight coupling — A must know B exists.

Blackboard

All agents read/write a shared workspace. Loosely coupled. Risks: write conflicts, stale reads, unclear ownership.

Event-Driven

Agents emit events; others subscribe. Highly decoupled. Harder to trace causation; needs a reliable event bus.

Documented Production Patterns

LangGraph (LangChain, 2024) formalized multi-agent communication as a directed graph. Edges between nodes represent message passing; each node is an agent or tool. The graph structure enforces who can talk to whom, eliminating the "agents talking to everyone" failure mode that caused redundant work in early LangChain chains.

Salesforce's Agentforce (launched September 2024) used a topic-routing approach: agents declared capability topics, and an orchestration layer matched incoming subtasks to the agent whose declared topics best fit. This removed the need for agents to know each other's identities — they communicated through the routing layer, not peer-to-peer.

Waymo's planning system uses a blackboard-style shared map state where perception agents, prediction agents, and planning agents all read a common world model. Strict versioning prevents agents from acting on stale sensor data — a critical constraint when vehicles are moving at highway speed.

Failure Mode: Conversational Loops

In early AutoGen deployments, two agents assigned to debate a topic would cycle indefinitely without a human-configured termination condition. Microsoft's fix was explicit: a max_consecutive_auto_reply parameter that forced a halt. Without termination logic, multi-agent conversations are inherently divergent.

Key Terms
Message SchemaThe agreed structure of a message — fields, types, required content — that enables agents to parse each other's outputs reliably.
Blackboard ArchitectureA shared workspace all agents can read and write; useful for loosely coupled coordination but requires conflict resolution.
Turn-Taking ProtocolRules governing which agent speaks next; prevents simultaneous conflicting actions and infinite loops.
Termination ConditionAn explicit signal or threshold that tells agents a conversation or subtask is complete, halting further processing.

Module 6 · Lesson 3

Parallelism, Specialization, and Failure Modes

The speed gains of running agents in parallel — and the compounding errors that can follow.
What happens when one agent in a parallel pipeline produces wrong output, and no one checks before the next step commits?

In February 2024, a British Columbia Civil Resolution Tribunal ruled against Air Canada after its chatbot — operating as a specialized subagent — gave a passenger incorrect information about bereavement fare refund policies. The tribunal held the airline responsible for the agent's output. This was not a multi-agent system in the technical sense, but the ruling established a precedent: automated agents acting on behalf of organizations create binding commitments, even when those agents produce incorrect outputs. In a parallel multi-agent pipeline, the surface area for such binding errors multiplies with each agent added.

Why Parallelism Is Appealing — and Dangerous

Parallel multi-agent execution reduces wall-clock time on tasks that have independent subtasks. A research pipeline might simultaneously run a literature-search agent, a data-retrieval agent, and a hypothesis-generation agent — completing in the time of the slowest, not the sum of all. This is the genuine productivity gain that systems like GPT-4 with tools and Claude's computer use were designed to exploit.

The danger: errors do not wait in a parallel system. If Agent A produces a flawed intermediate result and Agent B consumes it without verification before committing an irreversible action (sending an email, modifying a database, making a purchase), the error has propagated to the real world before any check occurs. In sequential systems, each step is at least a potential review point. In parallel systems, the review window closes simultaneously across all branches.

Documented Specialization Benefits

DeepMind's Gemini 1.5 Pro multi-agent research (2024) showed that routing long-document tasks to a dedicated retrieval subagent — rather than loading the full document into a single agent's context — improved accuracy on needle-in-a-haystack tasks by eliminating context-window noise. Specialization is not just an efficiency choice; it is an accuracy choice.

Imbue (formerly Generally Intelligent), 2024 demonstrated that coding agents with narrow, deeply tuned contexts for specific programming languages outperformed a single general agent across all languages. The tradeoff: each specialist had to be maintained separately, and routing errors (sending a Python task to the Ruby specialist) caused silent failures — the specialist would attempt the task and return plausible but wrong code.

Compounding Error Risk

In a 2024 analysis of multi-agent pipelines at Anthropic, researchers noted that in a five-agent sequential chain where each agent operates at 95% accuracy, the compound accuracy of the full pipeline is only 77%. Each additional agent multiplies the probability of cumulative failure. Parallelism does not reduce this — it accelerates it.

Key Failure Modes
Silent Routing Error

A task reaches the wrong specialist. The specialist returns plausible-looking but incorrect output. No error flag is raised.

Propagated Hallucination

One agent hallucinates a fact. Downstream agents treat it as ground truth. The error compounds through the pipeline.

Irreversible Commit

A subagent takes a real-world action (purchase, email, API write) before the orchestrator validates the triggering output.

Race Condition

Two parallel agents modify shared state simultaneously. Final state depends on which completed last — unpredictable and inconsistent.

Key Terms
Parallel ExecutionRunning multiple agents simultaneously on independent subtasks to reduce total wall-clock time.
Compounding Error RateThe multiplicative accumulation of individual agent error rates across a pipeline: (accuracy)^n for n agents.
Silent FailureAn error that produces plausible-looking output without raising an explicit flag, making it hard to detect downstream.
Irreversible Action GateA checkpoint requiring explicit validation before an agent can take actions that cannot be undone (writes, sends, purchases).
Design Principle

Anthropic's guidance on agentic systems recommends preferring reversible actions, requesting explicit confirmation for high-stakes steps, and erring on the side of doing less when uncertainty is high. This is doubly important in parallel pipelines where errors propagate without natural review points.

Module 6 · Lesson 4

Real-World Multi-Agent Deployments

Where these systems have actually shipped, what they accomplished, and what broke.
Beyond the demos and papers, which multi-agent systems have operated at real scale — and what did deployment reveal that lab tests missed?

In 2024, Scale AI deployed a multi-agent evaluation pipeline — codenamed SEAL — to benchmark frontier models against human expert performance. The system used a planning agent to decompose benchmark categories, specialist evaluation agents to run domain-specific tests, and an aggregation agent to compile leaderboard scores. Running this in parallel across dozens of models and categories, Scale reduced evaluation time from weeks to days. But deployment revealed an unexpected failure: specialist agents calibrated for one model's output format would misparse another model's outputs, returning silently wrong scores. The fix required adding a format-normalization preprocessing agent — an emergent architectural requirement that was not anticipated in the lab.

Five Deployed Systems, Five Lessons

1. Sierra AI's customer service agents (2024) — Bret Taylor's company deployed multi-agent systems for enterprise customer service at companies including SiriusXM and Sonos. Key finding: agents needed explicit "hand-off protocols" — structured messages that transferred customer context between agents with zero information loss. Informal hand-offs, even with well-designed prompts, caused agents to ask customers to repeat information they had already provided, degrading experience scores.

2. Harvey AI's legal research pipeline (2023–2024) — Harvey deployed a multi-agent legal research system where a case-law retrieval agent, a statutory interpretation agent, and a memo-drafting agent worked in sequence. Lawyers reported high accuracy on well-established law but systematic overconfidence on emerging legal areas where retrieval agents returned sparse results. Sparse retrieval was not flagged as low-confidence — it was passed downstream as equivalent to dense retrieval. The drafting agent had no way to signal epistemic uncertainty.

3. Google DeepMind's RT-2 Robot Control (2023) — RT-2 used a multi-step agent pipeline where a vision-language agent interpreted scenes and a motor-control agent executed actions. Deployment in physical environments revealed that the vision agent occasionally misidentified object positions by small margins that were harmless in text but caused robot arm misses in 3D space. The system had no error-recovery subagent — failures required human reset.

4. JPMorgan's DocLLM (2024) — JPMorgan deployed a document-processing multi-agent system where extraction agents, validation agents, and compliance agents processed financial documents in parallel. Reported outcome: 95% reduction in manual document review time. Unreported initially: the compliance agent had a higher false-positive rate on non-US document formats, flagging compliant European documents for human review — creating a new bottleneck that partially offset the automation gains.

5. Adept AI's ACT-2 (2024) — Adept demonstrated a multi-agent system for enterprise workflows — one agent navigating browser UIs, another reasoning over extracted data. Key deployment finding: the browser agent's page-understanding degraded significantly on dynamically rendered JavaScript-heavy pages compared to static HTML — a capability gap invisible in demo environments but prevalent in real enterprise software.

The Pattern Across All Five

Every deployed system above encountered a class of failure that was not visible in controlled testing: format mismatches, confidence miscalibration, environment drift, bottleneck shifting, and capability gaps revealed only by real data. Multi-agent systems are more brittle than they appear in demos because each agent's assumptions about its inputs were calibrated on test distributions that don't perfectly match production.

Evaluation Frameworks for Multi-Agent Systems

Standard single-agent benchmarks (MMLU, HumanEval, GSM8K) do not capture multi-agent dynamics. In 2024, researchers at Berkeley and Stanford proposed AgentBench — a benchmark specifically testing agents on multi-step, multi-tool tasks in realistic environments including web browsers, databases, and operating systems. Early results showed frontier models achieving only 30–40% task completion rates on complex multi-step sequences, compared to near-ceiling performance on static benchmarks.

Confidence MiscalibrationWhen an agent treats low-quality or sparse retrieved information with the same confidence as high-quality dense information, passing uncertainty downstream as fact.
Distribution ShiftThe gap between data distributions in testing environments and real production data; causes failures invisible in controlled evaluations.
AgentBenchA 2024 benchmark from Berkeley/Stanford designed specifically to evaluate agents on multi-step, multi-tool tasks in realistic environments.
Practitioner Insight

Across all documented deployments, the most reliable mitigation was the same: treat each agent-to-agent interface as an explicit API contract with schema validation, not an informal prompt-to-prompt handshake. When outputs are typed, validated, and explicitly structured, mismatches become detectable errors rather than silent failures.
