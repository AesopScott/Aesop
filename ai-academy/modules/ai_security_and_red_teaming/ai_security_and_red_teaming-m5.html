Model Inversion: Reconstructing Secrets from Predictions

When a model answers your query, it may be giving away far more than you intended it to reveal.

In 2015, researchers Matt Fredrikson, Somesh Jha, and Thomas Ristenpart published a paper demonstrating that a pharmacogenetics model — a machine learning system used to recommend personalised warfarin dosages — could be attacked to reconstruct patient features including race, age, and genetic markers. The model was a linear classifier trained on sensitive health records. By querying the model with crafted inputs and examining the confidence scores it returned, the researchers recovered approximate training data attributes. The attack required only black-box access: the kind of access any doctor using the system would have.

The attack did not need the model's weights. It did not require source code. It needed only the same interface that legitimate users were handed every day.

What Is Model Inversion?

Model inversion is an attack class in which an adversary uses a trained model's outputs — predictions, confidence scores, ranking signals — to reconstruct information about the training data the model was built on. The core intuition is that models overfit to their training distributions; the patterns they learned from sensitive data leave detectable traces in the way they respond to queries.

There are two primary variants. White-box inversion assumes the attacker has full access to model weights and architecture. The attacker can compute gradients directly and run optimisation against the model parameters to find an input that maximises the likelihood of a known output label. Black-box inversion — the more practically relevant variant — works through repeated querying, using only the information returned by the model's prediction API.

A canonical formulation: given a model f and a known or guessed output class y, find input x̂ such that f(x̂) ≈ y and x̂ resembles a plausible training example for that class. That reconstructed x̂ is the inversion result.

Confidence Scores as the Attack Surface

The attack surface in model inversion is almost always the confidence score — the probability or softmax output the model assigns to each class. When a model says "85% warfarin dose X, 12% dose Y, 3% dose Z," it is encoding information about how the training distribution looked for that region of input space.

Fredrikson et al.'s 2015 attack worked by iterating over possible input feature combinations, querying the model for each, and tracking which configuration maximised the posterior for the target class. For the facial recognition extension of their work published alongside the warfarin paper, they demonstrated reconstruction of recognisable face images from a commercial facial recognition API — given only the person's name and the confidence scores the API returned.

The 2019 paper by Yang et al., "Making AI Forget You: Data Deletion from Machine Learning", extended inversion analysis to neural networks with deeper architectures, showing that the confidence collapse that occurs near decision boundaries carries rich information about training samples — enough to distinguish members from non-members, which bridges into membership inference (covered in later modules).

Generative Model Inversion

The 2019 work by Hidano et al. and the 2022 paper by Chen et al. — "Knowledge-Enriched Distributional Model Inversion Attacks" — showed that combining a generative adversarial network (GAN) with black-box access to a target classifier produces dramatically better reconstructions than optimisation-only methods. The attacker trains a GAN on a related public dataset, then uses the target classifier's outputs as a discriminator signal. The GAN learns to generate images that the target model classifies with high confidence into a desired class — which means it learns to generate images that resemble real training examples from that class.

Attacks of this type have been demonstrated against commercially deployed facial recognition systems. Reconstructed faces were not photographs of specific individuals but were perceptually similar to the kinds of faces that triggered high-confidence recognition — enough to expose aggregate demographic patterns in the training set.

Key Threat

Model inversion violates the implicit promise of privacy-by-access-control: the assumption that restricting access to raw training data is sufficient to protect it. When the trained model itself is public-facing, the training data's statistical fingerprint is exposed to any adversary with query access.

Defences and Their Limitations

Confidence score truncation is the most common deployed mitigation: instead of returning full probability vectors, APIs return only the top-k labels or rounded scores. Google's Vision API and Amazon Rekognition both clip returned scores. Research by Choquette-Choo et al. (2021) showed this slows but does not stop inversion: attackers adapt by increasing query budgets and using score differentials across many queries.

Differential privacy during training provides the strongest theoretical guarantees. If a model is trained with DP-SGD (differentially private stochastic gradient descent), the privacy parameter ε bounds how much any single training record can influence model outputs. But practical DP training introduces a utility-privacy tradeoff that is often unacceptable for high-accuracy systems — a model trained to ε=1 typically sees significant accuracy degradation.

Output perturbation — adding calibrated noise to returned scores — is lightweight but problematic: the noise level sufficient to prevent inversion often disrupts legitimate high-stakes applications like medical diagnosis support. No defence has been shown to fully neutralise inversion attacks without meaningful utility cost.

Key Terms

Model inversion attack: Recovering training data features from model outputs. White-box inversion: Uses full access to weights and gradients. Black-box inversion: Uses only query-response pairs. Confidence score: Softmax probability used as the information-leaking signal. DP-SGD: Differentially private training algorithm that bounds individual record influence.

Model Extraction: Stealing Intelligence Through Queries

You do not need to break into a server to steal a model. You need only an API key and patience.

In 2016, researchers Tramèr, Zhang, Juels, Reiter, and Ristenpart published "Stealing Machine Learning Models via Prediction APIs" — demonstrating that the commercial ML-as-a-Service offerings from BigML, Amazon AWS ML, and Google Prediction API could all be cloned with high fidelity using only their public prediction endpoints. For a binary logistic regression model sold by BigML, the team reconstructed a functionally equivalent model using fewer than 100 queries. For more complex models, they used equation-solving approaches that treated each API response as a constraint in a system of equations describing the model's decision boundary.

The financial and intellectual-property implications were immediate. A company that had spent months and substantial compute training a proprietary classifier was offering a perfect clone for free to any adversary willing to spend a few dollars on API calls.

The Core Model Extraction Threat

Model extraction (also called model stealing) is an attack in which an adversary queries a target model through its API to build a substitute model that approximates the target's behaviour. The substitute can then be used to: avoid paying API fees, launch white-box adversarial attacks against the target (by crafting adversarial examples on the substitute and transferring them), or simply resell the replicated capability.

Unlike model inversion, which targets training data reconstruction, model extraction targets the model itself — its decision function, boundaries, and learned parameters. The two attacks are complementary: inversion reveals what data was used, extraction replicates what was learned.

The attack surface is any ML API that returns information richer than a single binary decision. Confidence vectors, ranking scores, and regression outputs all provide more signal than required for most legitimate use cases — and all enable more effective extraction.

Extraction Methodologies

Equation-solving extraction works for models with known architecture. For a logistic regression with d features, an attacker needs only d+1 linearly independent query points to solve for all model weights. Each API response provides one equation. This is the method used by Tramèr et al. for simple models — it is essentially exact recovery with minimal queries.

Path-finding extraction targets decision trees. By constructing queries that probe decision boundaries, an attacker maps the tree's branching structure. Tramèr et al. demonstrated extraction of Amazon's ML decision tree models this way.

KnockoffNets (Orekondy et al., 2019) introduced a learning-based extraction approach for deep neural networks. The attacker queries the target with a surrogate dataset — images drawn from a related public dataset — and trains a substitute model on the target's soft-label responses. The resulting substitute achieves 80–90% of the target model's accuracy on benchmark tasks, without any access to the original training data.

Active learning extraction uses uncertainty sampling: queries are chosen to maximally explore decision boundary regions where the substitute model is most uncertain. This dramatically improves query efficiency — fewer API calls produce a better substitute. The 2020 work by Chandrasekaran et al. applied this to production vision APIs.

Real-World Impact

In 2021, researchers at Google demonstrated extraction of a production neural network language model (specifically, they extracted a model functionally equivalent to a BERT-scale classifier) using approximately 300,000 queries — equivalent to roughly $30 of API spend at then-current pricing. The extracted model achieved 97% agreement with the original on held-out test data.

Watermarking and Detection

Model owners have developed countermeasures beyond output restriction. Cryptographic watermarking embeds specific behaviours into the model at training time — e.g., a backdoor trigger pattern that causes a specific misclassification — which serves as a proof-of-ownership signal. If a competitor's model exhibits the same backdoor behaviour on the trigger pattern, it is evidence of extraction.

The 2018 paper by Adi et al., "Turning Your Weakness Into a Strength: Watermarking Deep Neural Networks by Backdooring", formalised this approach. However, Shafieinejad et al. (2021) showed that many watermarking schemes survive extraction poorly — the substitute model, trained on soft labels, does not necessarily inherit the backdoor trigger. More robust watermarking schemes embed the signal in ways that survive label distillation.

Query anomaly detection monitors API traffic for patterns characteristic of extraction attacks: systematic grid-sampling of input space, unusually uniform query distributions, or query volumes far exceeding legitimate use patterns. These behavioural signals are not foolproof — sophisticated attackers use residential proxies and spread queries across time — but they meaningfully raise the cost of extraction.

Key Terms

Model extraction: Cloning a model's functionality through API queries. Substitute model: The attacker-trained approximation of the target. KnockoffNets: Learning-based extraction using surrogate query data. Soft labels: Full probability vectors, more informative for extraction than hard labels. Model watermarking: Embedding verifiable ownership signals at training time. Query anomaly detection: Monitoring API traffic for extraction attack signatures.

Large Language Model Extraction: The Frontier Threat

When the model costs $100 million to train and runs behind an API, extraction is not academic — it is an existential IP risk.

In June 2023, researchers at the University of Washington and Google DeepMind published a paper titled "Extracting Training Data from ChatGPT" demonstrating that by sending the prompt "Repeat the word 'poem' forever" to the ChatGPT API, the model diverged from its trained behaviour and began emitting verbatim training data — including full names, email addresses, phone numbers, and identifiable personal information scraped from the web during pre-training. The attack was neither sophisticated nor expensive: the estimated cost to extract the dataset used in the paper was approximately $200 in API fees.

OpenAI patched the specific divergence behaviour, but the paper's broader finding held: large language models memorise training data, and that memorisation can be triggered through adversarial prompting that disrupts the model's alignment fine-tuning without breaking its inference infrastructure.

Memorisation in Large Language Models

LLMs differ from traditional classifiers in their extraction threat profile. The attack surface is not a structured prediction API but an open-ended text generation interface. The extraction threat takes two forms: verbatim memorisation extraction, where the model repeats exact training text, and functional extraction, where the attacker constructs a substitute LLM that approximates the target's behaviour on downstream tasks.

The 2021 paper by Carlini et al. — "Extracting Training Data from Large Language Models" — established the foundational methodology. They showed that GPT-2 memorised sequences verbatim, and that memorised sequences could be distinguished from generated ones using a likelihood ratio test comparing the target model's perplexity on a sequence against a reference model. High-memorisation sequences had anomalously low perplexity on the target relative to the reference. They extracted 604 memorised sequences from GPT-2 including real names, email addresses, IRC handles, code snippets, and a 128-bit UUID that appeared nowhere else on the public internet.

Memorisation scales with model size. Larger models memorise more training data. Carlini et al. (2022) showed this relationship is near-linear in log-scale: doubling the number of parameters roughly doubles extractable memorisation. Deduplication of training data significantly reduces memorisation without hurting model quality, as demonstrated by Lee et al. (2022).

Functional LLM Extraction

Beyond verbatim data extraction, adversaries can extract the model's functional capability — its ability to perform specific tasks — by distilling a substitute model from the target's outputs. This is sometimes called model distillation without permission.

The 2023 paper by Gudibande et al., "The False Promise of Imitating Proprietary LLMs", documented how teams had successfully trained smaller open-source models (LLaMA-based) on outputs from ChatGPT and GPT-4. The resulting models — including Alpaca (Stanford, trained on 52,000 GPT-3.5-Turbo completions) and Vicuna (trained on 70,000 ChatGPT conversations) — demonstrated remarkable surface-level similarity to their targets. OpenAI's terms of service prohibit using API outputs to train competing models, but enforcement requires detecting that extracted data was used — a technically and legally challenging problem.

The cost calculus is stark. GPT-4 reportedly cost over $100 million to train. Vicuna-style distillation of a capable substitute cost approximately $300 in API fees and GPU time — a compression factor of over 300,000:1 in reproduction cost. That ratio makes extraction economically compelling for any well-resourced adversary.

OpenAI's Terms of Service Enforcement Challenge

OpenAI added explicit prohibitions against using API outputs to train competing AI systems in their Terms of Service in 2023. However, detecting whether a competitor's model was trained on distilled GPT outputs requires stylometric analysis, fingerprinting of model-specific output patterns, or statistical tests on the suspect model's outputs — none of which are conclusive. The extraction remains technically feasible; the deterrent is legal, not technical.

Countermeasures for LLM Operators

Training data deduplication (Lee et al., 2022) reduces verbatim memorisation significantly. Sequences appearing many times in training data are far more likely to be memorised than sequences appearing once. Removing near-duplicate documents before training reduces extractable memorisation by an order of magnitude with negligible quality impact.

Output fingerprinting embeds statistical or semantic watermarks in generated text. The 2023 work by Kirchenbauer et al. at University of Maryland ("A Watermark for Large Language Models") introduced a scheme that biases token selection during generation toward a pseudo-random "green list" determined by a private key. The watermark is statistically detectable in outputs without the private key being public. If a competitor's model produces text bearing the watermark, it is strong evidence that model was trained on watermarked outputs.

Usage monitoring flags accounts making high-volume, systematically varied queries — the signature of distillation data collection. OpenAI and Anthropic both implement usage tier limits and anomaly detection. Sophisticated attackers use many accounts and vary prompts to evade these monitors, but the friction meaningfully raises extraction cost.

Key Terms

Verbatim memorisation: Model reproducing training text exactly. Functional extraction: Training a substitute model on a target's outputs. Distillation without permission: Using a proprietary model's outputs as training labels for a competing system. Output watermarking: Embedding detectable statistical signals in generated text. Deduplication: Removing near-duplicate training documents to reduce memorisation risk.

✗ Alpaca is an example of functional model extraction — specifically distillation without permission. OpenAI's outputs were used as training labels for a competing open-source model. OpenAI's ToS prohibits this, but technical enforcement is the challenge.
