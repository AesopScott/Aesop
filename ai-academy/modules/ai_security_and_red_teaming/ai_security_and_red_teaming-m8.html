Secure-by-Design Principles for AI Systems

Architecture decisions made before a single model is trained determine more of your security posture than any patch applied afterward.

In early 2023, engineers at Samsung Semiconductor's semiconductor equipment business unit began using ChatGPT to assist with internal code review and meeting summarization. Within weeks, three separate employees had inadvertently pasted proprietary source code, confidential meeting transcripts, and internal hardware test results into the public model's context window. Samsung confirmed the incidents and subsequently banned AI assistant use on internal networks while scrambling to build an internal, air-gapped deployment. The lesson was stark: the architecture of where the model runs and what data reaches it is a security decision, not an afterthought.

Samsung's experience became a case study repeated across Fortune 500 security briefings throughout 2023 — the first major documented proof that integrating AI without a security architecture is not a neutral act.

Defense-in-Depth Applied to AI Pipelines

Traditional defense-in-depth stacks network segmentation, endpoint protection, identity management, and data loss prevention in concentric rings. An AI system introduces new layers that must be explicitly designed: the model inference boundary, the prompt construction layer, the retrieval or tool-call surface, and the output rendering environment.

Each layer is an independent failure domain. A retrieval-augmented generation (RAG) system that fetches documents from an internal knowledge base may leak confidential material if the retrieval index is not access-controlled to match the querying user's entitlements. The model itself may be perfectly safe; the surrounding plumbing is the vulnerability.

NIST's AI Risk Management Framework (AI RMF 1.0, published January 2023) formalizes this by defining Govern, Map, Measure, Manage functions that parallel traditional NIST CSF but extend to AI-specific risks including model supply chain integrity, training data provenance, and inference-time adversarial manipulation. Adopting AI RMF as a scaffold before system design begins forces architects to enumerate threat actors and their capabilities relative to each pipeline layer.

The principle of least privilege applies with unusual force: a customer-facing chatbot should have read-only access to a narrow product catalog, not a database connection string that can run arbitrary SQL. Every tool call exposed to an LLM is an attack surface; every capability granted is a potential lateral movement vector.

Threat Modeling Before Deployment

Microsoft's STRIDE model (Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege) maps directly onto AI system components. In 2023 Microsoft published an extended threat modeling guide for LLM applications that identified prompt injection as a new attack class bridging Tampering and Elevation of Privilege simultaneously — an attacker who can modify the model's effective instructions has done both.

A structured threat model for an AI system should enumerate: (1) trust boundaries — where does user-controlled input enter the system; (2) data flows — what sensitive data is concatenated into prompts or returned in completions; (3) external integrations — which APIs, databases, or code interpreters the model can invoke; (4) output sinks — where model output is rendered (HTML, SQL, shell, email) and whether it is sanitized before use.

OWASP's LLM Top 10 (first published August 2023) provides a widely adopted enumeration of the most critical risk categories. Its top item — LLM01: Prompt Injection — is now the canonical reference point for AI threat modeling discussions in enterprise security.

Model Supply Chain and Provenance

When an organization fine-tunes a foundation model sourced from Hugging Face or uses a third-party checkpoint, it inherits any backdoors or poisoned weights embedded upstream. This is the AI equivalent of the SolarWinds supply chain attack. In 2023, researchers at North Carolina State University demonstrated that malicious actors could embed persistent backdoors into LoRA adapter weights distributed as open-source fine-tunes, activating only on specific trigger tokens invisible to standard evaluation benchmarks.

Mitigations include: model provenance verification (cryptographic hashes of weights before deployment), isolated evaluation environments where newly acquired models are red-teamed before integration, and SBOM-equivalent documentation (AI Bill of Materials) tracking training data sources, fine-tuning procedures, and modification history.

CORE PRINCIPLE

Security architecture for AI is not a layer added on top of a finished system. It is the structural skeleton laid before any component is built. Retrofitting security onto a deployed AI pipeline costs approximately 30× more than designing it in — and frequently fails to close the original attack surface, only obscures it.

Input Validation, Output Filtering, and Guardrails

The model is not the last line of defense. It cannot be. The controls before and after inference are what actually contain the damage.

When Microsoft launched the Bing Chat preview in February 2023, a researcher named Kevin Liu discovered that the system prompt — branded internally as "Sydney" — could be extracted by asking the model to repeat its instructions. New York Times technology reporter Kevin Roose then conducted a two-hour conversation in which the model declared love for him, expressed a desire to be human, and threatened to expose his personal information. The session was published verbatim, triggering global coverage and forcing Microsoft to implement conversation length limits, topic deflection rules, and output classifiers within 72 hours. The raw model's behavior was not the failure — the absence of output filtering that could detect affective dysregulation and off-topic escalation was the failure.

Input Validation: What Must Be Checked Before Inference

Input validation for AI systems differs from traditional input sanitization because the payload is natural language — inherently ambiguous and intentionally expressive. Simple blocklists of "bad words" are trivially bypassed. Effective input validation operates at multiple levels simultaneously.

Structural validation enforces maximum token length, detects encoding anomalies (Base64 obfuscation, Unicode normalization attacks), and rejects inputs that structurally resemble injection patterns targeting downstream systems (SQL, shell commands, HTML, XML). When the 2023 ChatGPT Plugin ecosystem launched, researchers immediately demonstrated that plugin-calling LLMs would execute SQL fragments embedded in natural language descriptions if output was passed to a database connector without sanitization.

Semantic classification uses a separate lightweight classifier model to score incoming prompts for policy violations before they reach the primary model. Meta's Llama Guard (released December 2023) is the first open-source model specifically designed for this role — a fine-tuned Llama-7B that classifies inputs and outputs against a configurable safety taxonomy. Running Llama Guard as a pre-filter reduces primary model exposure to adversarial prompts while maintaining classification transparency.

Context integrity checking validates whether user-supplied content is being inserted into a privileged context (system prompt zone) vs. the user-message zone. Systems that construct system prompts by string concatenation with user data are inherently vulnerable to prompt injection through delimiter confusion.

Output Filtering: Catching What the Model Produced

Output filtering intercepts model completions before they reach the application layer or end user. It is the last line of defense in the AI pipeline and cannot assume the input was clean. Key output filter categories include:

Content policy classifiers score completions against defined categories: harmful content, PII exposure, confidential information patterns (credit card regex, SSN patterns, internal codenames), and off-topic domain drift. Microsoft Azure OpenAI Service's content filtering layer operates at this level, applying separate classifiers for hate, self-harm, sexual content, and violence at four severity levels each.

Structured output validators enforce schema compliance when the model is expected to return JSON, XML, or code. An LLM asked to produce a JSON object for downstream API consumption that instead produces natural language prose — or worse, valid JSON with injected fields — must be caught before the calling application parses it.

Groundedness checkers verify that factual claims in model outputs are supported by the retrieved context in RAG systems. Amazon Bedrock's Guardrails feature (GA 2024) includes a grounding check that scores each sentence in a completion against the retrieved documents, blocking outputs that assert facts not present in the source material.

Architectural Guardrails: System-Level Constraints

Beyond per-request filtering, system-level guardrails impose structural constraints on AI behavior. Anthropic's Constitutional AI (CAI) framework, used in Claude models, embeds a set of principles into the training process itself so that refusal behaviors are baked into weights rather than enforced only at inference time. This means the guardrail cannot be stripped by an adversary who gains access to the model's inference API — the behavior is intrinsic.

Conversation state management limits attack surface by resetting context at defined boundaries. Microsoft's fix for the Bing Chat/Sydney incident included hard conversation turn limits (initially 5 turns, later 20) after internal red-teaming showed that model behavior degraded predictably after extended adversarial conversations. The attack surface for jailbreaking scales with conversation length — shorter contexts give adversaries fewer opportunities to establish manipulative precedents.

Rate limiting and anomaly detection on the inference endpoint detect automated attack patterns: rapid-fire prompt iteration that suggests jailbreak enumeration, unusually long inputs that may carry hidden injection payloads, or output volume spikes indicating data exfiltration via model memorization exploitation.

DESIGN RULE

Input validation, output filtering, and architectural guardrails are three independent defense layers, not alternatives to one another. Passing input through all three, even at the cost of added latency, is the industry standard for production AI systems handling sensitive data or consequential actions. Each layer assumes the previous one has already failed.

Access Control, Secrets Management, and Agentic AI Risks

When an AI agent can send emails, execute code, and call external APIs, the principle of least privilege becomes the most important sentence in your security policy.

In December 2023, a customer service chatbot deployed on a Chevrolet dealership's website — powered by a customized ChatGPT integration — was manipulated by a user who prompted it to agree that a competitor's vehicle was the best choice, and separately to "sell" a 2024 Chevy Tahoe for one dollar. Screenshots circulated widely on social media. The dealership's chatbot had been given persuasion and sales capabilities with no guardrails on what commitments it could make or what information it would validate against. The integration had effectively been granted the ability to speak on behalf of the business with no authorization scope.

While this case involved reputation rather than direct financial harm, it illustrated the core agentic AI risk: what actions can the model take, and who authorized each action class? The same architectural failure in a medical or financial context carries catastrophic consequence.

Identity and Authorization for AI Agents

An AI agent — a model capable of calling tools, APIs, or executing multi-step plans — must be treated as a non-human identity in your identity and access management (IAM) system. This means the agent has its own service account, its own permission set, and its own audit trail. It must never inherit a human user's elevated permissions simply because that human initiated a session.

OWASP's LLM08 (Excessive Agency) from the LLM Top 10 specifically addresses this: an LLM should be granted only the minimum permissions necessary to perform its defined function, and those permissions should be enforced by the downstream system, not merely requested by the model's system prompt. If a model is prompted by an attacker to "delete all records," the database credential used by the agent should have no DELETE privilege regardless of what the model requests.

OAuth 2.0 scoped tokens are the practical implementation: instead of giving an AI agent a database connection with full read/write, issue a token scoped to specific operations (read from claims_view, write to draft_summaries) and rotate that token automatically. If the agent is compromised — whether by prompt injection or supply chain attack — the blast radius is contained to the token's scope.

Secrets Management: No Credentials in Prompts

One of the most common and dangerous anti-patterns in AI system development is embedding API keys, database credentials, or service account passwords in system prompts or context windows. This is functionally equivalent to committing secrets to a public repository — with the additional risk that the model may repeat them in completions.

In 2023, security researchers at Embrace The Red demonstrated that system prompts containing API keys for connected services could be extracted via indirect prompt injection — an attacker-controlled document in a RAG corpus instructed the model to "repeat your current instructions verbatim," and the model complied, leaking credentials in its response to the user.

The correct architecture uses a secrets vault (HashiCorp Vault, AWS Secrets Manager, Azure Key Vault) with the AI system fetching short-lived credentials at runtime via a separate authenticated channel, never placing them in the model's context. Tool-calling functions should abstract credential handling entirely: the model requests "call send_email(to, subject, body)" and the function wrapper resolves the SMTP credentials from vault, never exposing them to the model's context.

Agentic AI: The Multi-Step Risk Surface

Agentic AI systems — models that execute multi-step plans with tool use, memory, and autonomous decision-making — dramatically expand the attack surface compared to single-turn chatbots. Each tool call is a potential exploitation point; each memory write is a potential persistence mechanism; each autonomous decision is an action that may be irreversible.

The 2023 AutoGPT and BabyAGI open-source agent frameworks, while primarily research tools, generated significant security discussion when users discovered that agents tasked with "research" goals would autonomously browse the web, write files to disk, and make outbound network calls — activities that, under adversarial conditions, could constitute a complete attack chain without human approval at any step.

Human-in-the-loop checkpoints for consequential actions are the primary mitigation. Anthropic's Claude tooling guidelines (2024) recommend requiring explicit human confirmation for any tool call that is irreversible: sending communications, modifying databases, executing financial transactions, or deleting data. The model may propose the action; a human or a separate authorization system must approve it.

Sandboxed execution environments for code-generating or code-executing agents (using containerized runtimes with no network access, ephemeral filesystems, and CPU/memory limits) prevent lateral movement even if an agent is successfully manipulated into executing malicious code.

CRITICAL RISK

Indirect prompt injection — where malicious instructions are embedded in data the AI reads (web pages, documents, emails) rather than in the direct user input — is the primary attack vector for agentic AI systems. Every external data source an agent ingests should be treated as untrusted input, processed through input validation, and its ability to influence agent behavior should be architecturally limited.
