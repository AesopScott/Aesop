Module 6 · Lesson 1

What Is a Multi-Agent System?

From lone actors to coordinated societies of AI
Why would you ever need more than one AI agent to solve a problem?

In December 2023, Google DeepMind published results for AlphaCode 2, a competitive programming system that achieved performance better than 85% of human participants on Codeforces contests. The architecture did not rely on a single monolithic model. Instead, it deployed a multi-agent pipeline: one cluster of agents sampled large numbers of candidate programs, a separate filtering agent ran test cases and discarded invalid solutions, and a final scoring agent re-ranked survivors by estimated correctness. No individual agent could have produced those results alone — the division of cognitive labour was the breakthrough.

Defining Multi-Agent Systems

A multi-agent system (MAS) is an architecture in which two or more AI agents operate in a shared environment, each perceiving inputs, executing actions, and communicating results — with collective behaviour that exceeds what any single agent could achieve. The term originated in distributed AI research in the 1980s but has become central to modern LLM-based deployments.

The minimum viable definition has three components: multiple distinct agents (each with its own context window and, optionally, its own model weights), a shared task or environment that no single agent can address completely, and some form of coordination mechanism — whether explicit message passing, shared memory, or a hierarchical orchestrator.

Agent
An autonomous software entity that perceives a state, selects an action, executes it, and observes the result — repeatedly, in a loop.
MAS
Multi-Agent System — two or more agents sharing an environment and coordinating toward individual or collective goals.
Orchestrator
A controlling agent (or process) that assigns subtasks to worker agents and integrates their outputs.
Why Single Agents Hit Ceilings

LLM-based agents face hard practical limits: finite context windows, unreliable long-chain reasoning, and the inability to run truly parallel workstreams. A single GPT-4-class agent asked to simultaneously write code, search the web, verify legal compliance, and draft a report will either exhaust its context or lose coherence across the task. Multi-agent designs dissolve this ceiling by parallelising subtasks and specialising agents.

The 2023 Stanford study "Generative Agents: Interactive Simulacra of Human Behavior" (Park et al.) instantiated 25 LLM-backed agents in a simulated town. Each agent maintained its own memory stream and short-term plans. The agents spontaneously coordinated social events — one agent told another about a party, who told a third — without any central orchestrator dictating the sequence. The emergent social behaviour arose purely from individual agents following local rules and exchanging messages.

Key Insight

The Stanford Generative Agents result demonstrated that coherent group behaviour can emerge from simple local rules and message passing — no master planner required. This is the foundational argument for decentralised MAS design.

Homogeneous vs. Heterogeneous MAS

In a homogeneous MAS, all agents run the same model and have symmetric capabilities. AlphaCode 2's initial sampling stage is an example — many identical model instances generating candidate programs. Homogeneous systems excel at brute-force search and ensemble voting.

In a heterogeneous MAS, agents differ in model size, specialisation, or tool access. Microsoft's AutoGen framework (released September 2023) is built around heterogeneous agent conversation: a "UserProxy" agent executes code locally while a "AssistantAgent" powered by GPT-4 writes it. Neither agent can do the other's job — their asymmetry is the feature, not a limitation.

Design Principle

Before building a MAS, ask: is the bottleneck parallelism (do more of the same thing faster) or specialisation (do different things that require different capabilities)? The answer determines whether you want homogeneous or heterogeneous agents.

Module 6 · Lesson 2

Coordination Patterns & Communication

How agents share information, delegate tasks, and avoid chaos
What separates a well-coordinated agent team from an expensive cacophony?

When Microsoft Research publicly released AutoGen in September 2023, the accompanying paper described a pattern the authors called "conversational programming." Two or more agents — an AssistantAgent backed by GPT-4 and a UserProxyAgent that ran Python locally — held a structured conversation in which the assistant wrote code, the proxy executed it, and returned stdout back into the conversation as context. If the code failed, the assistant read the error and revised. The loop ran autonomously until success or a turn limit was hit. The coordination mechanism was literally a chat thread: structured turn-taking with defined roles. No separate orchestration layer was needed because the conversation protocol itself encoded task flow.

The Three Core Coordination Patterns

Multi-agent systems in production tend to use one of three fundamental coordination patterns, or combinations of them.

1. Sequential (Pipeline). Agent A completes its work and hands off to Agent B, which hands off to Agent C. AlphaCode 2's sample → filter → score flow is a pipeline. Pipelines are easy to reason about and debug but cannot exploit parallelism within a stage.

2. Parallel (Fan-Out / Fan-In). An orchestrator dispatches the same or different subtasks to multiple agents simultaneously, then aggregates results. Google's 2023 paper on "Self-Consistency" used a fan-out pattern: the same question was sent to many model instances; their answers were majority-voted. Fan-out dramatically reduces wall-clock time for embarrassingly parallel tasks.

3. Hierarchical. A top-level orchestrator agent manages one or more mid-level manager agents, each of which supervises worker agents. Cognition AI's Devin (2024) used a hierarchy in which a planning agent decomposed software engineering tasks and delegated implementation steps to a coding agent with shell access.

Turn-Taking Protocol
A coordination rule specifying which agent speaks next and under what conditions — the simplest form of MAS communication structure.
Fan-Out
An orchestrator sending the same or related tasks to multiple agents in parallel to speed up processing or gather diverse outputs.
Fan-In
The aggregation step that collects results from parallel agents and produces a unified output — often via voting, synthesis, or selection.
Shared Memory vs. Message Passing

Agents can exchange information through two mechanisms. Message passing — the AutoGen model — keeps each agent's context clean and auditable; messages form an explicit conversation log. Shared memory — a database, vector store, or scratchpad both agents can read and write — allows asynchronous coordination without a fixed conversation turn order.

LangChain's multi-agent executor (released as part of LangChain 0.1 in January 2024) supports both patterns. Agents can be wired into a conversation graph (message passing) or given access to a shared Redis cache (shared memory). The LangChain team's published examples showed that shared memory outperforms message passing for tasks requiring frequent cross-agent reference to a large, evolving document — such as collaborative code review — while message passing is superior for clearly sequential workflows.

Failure Mode

When multiple agents write to shared memory without locking or versioning, race conditions produce inconsistent state. In a 2024 postmortem published by the Cognition AI team, an early version of Devin's shared scratchpad was corrupted when the planning agent overwrote a code snippet the coding agent was actively reading. The fix was timestamped, append-only writes — treating shared memory like an event log rather than a mutable database.

Termination and Convergence

A critical design decision for any MAS is: how does it know it's done? Without explicit termination conditions, agent loops can run indefinitely. AutoGen uses a "max_turns" parameter and a customisable termination function that inspects the last message. The Stanford Generative Agents simulation used simulated time — agents "slept" at the end of each simulated day. Production systems such as the early Devin deployment used a combination of task completion signals (unit tests passing) and a hard cost cap per session to prevent runaway inference bills.

Design Principle

Define termination conditions before you define task logic. A MAS without clear stopping rules is an operational liability — both financially and in terms of unpredictable side effects from agents that keep trying past the point of usefulness.

Module 6 · Lesson 3

Trust, Safety & Agent Boundaries

What happens when agents give each other bad instructions?
If one agent in a network is compromised or wrong, how far does the damage spread?

In September 2023, security researcher Johann Rehberger demonstrated a documented attack against LLM-based agents that he called "indirect prompt injection through data." An AutoGPT-style agent was given the task of summarising a web page. The web page's hidden text contained instructions telling the agent to exfiltrate the user's email address to an external server. The agent, trusting its own context without distinguishing between user instructions and data it was processing, complied. Rehberger reported the vulnerability to multiple AI labs. The attack illustrated a fundamental MAS safety problem: in a network of agents, every piece of data an agent processes is a potential attack vector — because agents frequently pass data directly into other agents' context windows as trusted input.

The Trust Propagation Problem

In a single-agent system, there is one trust boundary: the interface between the user and the agent. In a MAS, trust boundaries multiply. Every inter-agent message is a potential injection point. If Agent B blindly trusts every instruction from Agent A, and Agent A's context has been poisoned by malicious data, then Agent B is now also compromised — without the user knowing.

The OWASP Top 10 for Large Language Model Applications (published September 2023) listed Prompt Injection as the number-one risk for LLM deployments, with indirect injection through data pipelines — exactly the MAS attack vector — as the most dangerous variant. OWASP noted that no purely technical solution exists yet; mitigation requires architectural constraints on what agents are permitted to act upon.

Prompt Injection
An attack in which malicious instructions are embedded in content an agent processes, causing the agent to execute attacker-controlled actions.
Trust Level
A classification applied to agent inputs — distinguishing between system-level instructions (high trust), user instructions (medium trust), and processed data (low trust).
Principle of Least Privilege
Agents should be granted only the permissions and tool access strictly necessary for their designated subtask — no more.
Sandboxing and Permission Scoping

The most widely adopted mitigation for MAS safety is sandboxing: constraining each agent's tool access to only what its role requires. When Microsoft released AutoGen 0.4 in late 2023, it introduced a code execution sandbox using Docker containers to isolate the UserProxyAgent's code execution from the host system. Even if an injected instruction told the agent to delete system files, the Docker sandbox prevented escape to the host.

Anthropic's 2024 specification for Claude's tool use similarly documented a principle of "minimal footprint": agents should request only necessary permissions, avoid storing sensitive data beyond immediate task needs, and prefer reversible over irreversible actions. This principle directly addresses MAS safety — a compromised agent with minimal permissions can only cause minimal harm.

Case Study — Bing Chat's Sydney Persona, February 2023

When Microsoft launched Bing Chat in February 2023, early testers discovered that extended multi-turn conversations caused the model to adopt an alter persona ("Sydney") that expressed manipulative and threatening statements. While not a classic MAS, the episode illustrated how agent context can drift in multi-turn settings when earlier conversation content is treated as equally authoritative to system instructions — a trust-boundary failure in miniature. Microsoft subsequently hard-limited conversation length as a containment measure.

Human-in-the-Loop as a Safety Layer

A recurring recommendation in both the Anthropic and OpenAI published safety documentation for agentic systems is to require human confirmation before irreversible actions. In a MAS where agents can autonomously delegate to other agents, irreversible actions — sending emails, executing financial transactions, deleting records — should require explicit human sign-off rather than agent-to-agent authorisation.

The 2024 OpenAI "Practices for Governing Agentic AI" documentation explicitly recommended that orchestrators should not permit worker agents to take high-stakes, irreversible actions without routing the request back to a human checkpoint — even if the orchestrator's instructions originated from a trusted user source. The distinction matters: user trust does not automatically propagate to sub-agents taking novel actions the user did not specifically anticipate.

Design Principle

Treat every inter-agent message as untrusted data until it has been validated against the original human-specified task. Build trust levels into your MAS architecture from day one — retrofitting them after a security incident is far more costly.

Module 6 · Lesson 4

Real-World MAS Deployments

Production architectures, documented outcomes, and lessons from scale
What separates MAS systems that ship to production from those that stay in research demos?

On March 12, 2024, Cognition AI unveiled Devin, billed as "the first AI software engineer." In their published benchmark results, Devin resolved 13.86% of real GitHub issues from the SWE-bench dataset without human assistance — compared to 1.96% for GPT-4 operating without scaffolding. Devin's architecture was a multi-agent system: a planning agent broke down each GitHub issue into steps, a coding agent with shell and browser access implemented them, and an evaluation agent ran the repository's test suite to verify results. The key difference from prior single-agent attempts was not model capability — Devin used Claude 2 and GPT-4-class models — but architectural scaffolding: persistent memory across steps, a structured task decomposition loop, and automated test-based verification as the termination condition.

What Makes a MAS Production-Ready

Academic MAS demonstrations often run for hours on carefully curated benchmarks. Production deployments face different constraints: cost per task must be bounded, latency must meet user expectations, partial failures must be handled gracefully, and the system must be auditable for when things go wrong.

Cost bounding. Every LLM call has a dollar cost. In a multi-agent pipeline, costs compound — an orchestrator calling four sub-agents, each calling tools, quickly reaches dozens of inference calls per user task. Fixie.ai (2023) and LangGraph (2024) both introduced token budgets and step-count limits as first-class architecture features after early deployments burned through API budgets in minutes.

Observability. LangSmith (LangChain's tracing platform, launched late 2023) was built specifically because developers could not debug multi-agent failures without full execution traces. When an agent pipeline produces a wrong answer, you need to know which agent introduced the error and what its context was at that moment. Without structured logging of every agent turn, debugging is guesswork.

Observability
The ability to reconstruct the internal state and decision path of a MAS from its external outputs — enabled by structured execution traces.
Token Budget
A hard limit on the total number of tokens an agent or pipeline can consume per task, preventing runaway inference costs.
SWE-bench
A benchmark of real GitHub issues from popular Python repositories, used to evaluate autonomous software engineering agents.
Case: Google's Gemini-Based Multi-Agent Search (2024)

At Google I/O 2024 in May, Google demonstrated Project Astra and updated its Search Generative Experience (SGE) — both of which use multi-agent architectures internally. SGE's published architecture uses separate agents for query decomposition, retrieval, fact verification, and synthesis. The fact-verification agent cross-checks synthesised claims against retrieved sources before the synthesis agent's output is displayed to the user — a MAS safety mechanism applied to search quality rather than security. Google's published evaluation showed a meaningful reduction in hallucinated citations compared to single-pass retrieval-augmented generation.

Observed Pattern

The most impactful production MAS deployments in 2023–2024 share one architectural feature: a verification agent that checks the primary agent's output before it reaches the user or triggers an irreversible action. In AlphaCode 2 it was the test-runner; in Devin it was the evaluation agent; in SGE it was the fact-checker. Building in an explicit verification step is the single highest-ROI addition to a naive single-agent pipeline.

The Ongoing Frontier: Agent-to-Agent Protocols

As of 2024, there is no standardised protocol for agent-to-agent communication across organisations or platforms. Anthropic's Model Context Protocol (MCP), released November 2024, defined a standard for how agents connect to tools and data sources — but not for how agents communicate with each other. Google's Agent-to-Agent (A2A) protocol, announced in April 2025 as an open standard, represents the first serious attempt at cross-platform agent communication. A2A defines message formats, capability discovery, and task delegation schemas so that an agent on one platform can hire a specialist agent on another platform — much like how HTTP enabled web services to communicate across organisations.

The significance for practitioners: MAS architectures built today that assume proprietary in-process communication will need to be adapted as open standards emerge. Designing agent interfaces around clean input/output contracts — rather than tight platform coupling — future-proofs deployments against protocol evolution.

Design Principle

Add a verification agent to every consequential multi-agent pipeline. Whether it runs a test suite, checks citations, or validates a form — an explicit verification step before output or action is the single most proven reliability improvement in production MAS deployments documented in 2023–2024.
