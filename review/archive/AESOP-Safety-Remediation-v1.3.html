Age Gate & Parental Consent Mechanism

📐 D1 COPPA Core ⏱ 4–8 hrs ⚡ Removes D1 Override
›
Problem

No age screening exists at platform entry. Users under 13 can register and access content without parental consent, directly violating COPPA §312.5. This is the primary trigger for the D1 override.

Solution

Implement a two-path entry gate on registration and first module access:

  • Path A (13+): Standard registration with DOB confirmation. If DOB entered is under 13, route to Path B.
  • Path B (Under 13): Collect parent/guardian email. Send verifiable parental consent (VPC) email via PHPMailer. Gate module access until consent email link is clicked.
  • Store consent record in Firestore with timestamp, parent email, and child UID.
Implementation Steps
  • Add DOB field to Firebase Auth registration flow (existing auth modal)
  • Build consent-gate.html — parent email collection + consent email trigger
  • Build consent-confirm.php — link target that writes Firestore record and redirects
  • Add Firestore rule: module pages beyond Module 0 require consentVerified: true in user doc
  • Update Module 0 localStorage gate to also check Firebase consent flag
Expected Score Impact
D1: +10–14 pts (override cleared) D5: +2–3 pts
P1 — CRITICAL

Privacy Notice for Children (COPPA-Compliant)

📐 D1 + D4 ⏱ 2–3 hrs ⚡ Addresses D1 + D4 simultaneously
›
Problem

No privacy notice exists anywhere on the platform. COPPA §312.4 requires a clear, prominent notice of data collection practices before collecting any personal information from children. D4 (Transparency) scored 3/20 primarily due to this absence.

Solution

Build a plain-language privacy notice page targeted at both parents and children:

  • What data is collected (name, email, usage data via Firebase Analytics)
  • How it is used (account access, progress tracking, no advertising)
  • Who it is shared with (Firebase/Google only, no third-party advertising)
  • Parental rights: access, deletion, withdrawal of consent
  • Contact: [email protected]
Implementation Steps
  • Build privacy.html — deploy to aesopacademy.org root
  • Link in footer of every page and in the consent gate email
  • Add inline "What we collect" summary to the registration modal (2–3 sentences max)
  • Reference privacy.html in the consent email template
Expected Score Impact
D1: +3–4 pts D4: +6–8 pts
P2 — HIGH

Firebase Analytics — Enable COPPA Mode

📐 D3 Data Security ⏱ 1–2 hrs ⚡ Code + Firebase Console change
›
Problem

Firebase Analytics is active on the platform without COPPA mode enabled. By default, Firebase Analytics collects device identifiers and advertising IDs. This is non-compliant for platforms serving children under 13.

Solution

Enable COPPA mode on Firebase Analytics to disable advertising ID collection and restrict data use for under-13 users. This is a single flag, not a rebuild.

Implementation Steps
// In your Firebase init (every page that loads Firebase): firebase.analytics().setAnalyticsCollectionEnabled(true); // After user auth, if user is under 13: firebase.analytics().setUserProperties({ 'is_child_directed': 'true' }); // OR — simplest: set globally in Firebase Console // Firebase Console → Analytics → Settings → // Data Privacy → Enable "Child-directed treatment"
  • Recommended: enable at the Firebase Console level (applies globally, no per-page code needed)
  • Verify in Firebase DebugView that advertising_id is null after enabling
  • Add note to privacy.html: "We use Firebase Analytics in child-directed mode. No advertising IDs are collected."
Expected Score Impact
D3: +3–4 pts D5: +1–2 pts
P2 — HIGH

AI Disclosure — Inform Users Content is AI-Generated

📐 D4 Transparency ⏱ 1 hr ⚡ UI/copy change only
›
Problem

No disclosure exists that LIBREX/AESOP story content, quiz items, and lab prompts are AI-generated. Emerging state laws (KOSA, California AB 2355) and FTC guidance increasingly require AI disclosure in child-directed platforms.

Solution

Add a brief, age-appropriate AI disclosure in two locations:

  • Module 0 orientation: One paragraph explaining that LIBREX is an AI character and that story content is AI-generated to make learning interactive.
  • Privacy notice: Section explaining AI content generation, what data is sent to AI services, and what is not stored.
Sample Copy (Module 0)
LIBREX is an AI-powered learning guide. The stories, questions, and examples in this course are created with AI to make learning more engaging. LIBREX doesn't remember you between sessions and doesn't store your conversations.

P3 — MEDIUM

Parent Dashboard — Data Access & Deletion Portal

📐 D1 + D4 ⏱ 4–6 hrs ⚡ New page + Firebase Admin calls
›
Problem

COPPA §312.6 requires operators to provide parents the ability to review, correct, and delete their child's personal information upon request. No mechanism exists for this currently.

Solution

Build a minimal parent portal accessible via email link (no separate auth needed). Parent enters the email used during consent verification — system looks up child account and presents data summary with delete option.

Minimum Viable Scope
  • Build parent-portal.html — parent email input → lookup consent record → show child display name + progress summary
  • Add "Delete my child's account" button → calls Firebase Admin SDK (server-side via PHP) to delete Auth user + Firestore records
  • Send confirmation email to parent on deletion
  • Link in privacy.html: "Parents: click here to access or delete your child's data"
Expected Score Impact
D1: +2–3 pts D4: +3–4 pts

📐 D3 Data Security ⏱ 2–3 hrs ⚡ Documentation + config review
›
Problem

Two third-party data processors are active: fal.ai (video generation — prompt data transmitted) and Firebase (auth, Firestore, Analytics). Data retention policies and COPPA agreements with these vendors are unconfirmed.

Solution

  • fal.ai: Review fal.ai terms of service for data retention on prompt inputs. If user-generated prompts could contain PII (child's name, etc.), consider stripping before transmission or routing through claude-proxy.php as a sanitization layer.
  • Document both integrations in privacy.html under "Third-Party Services"

P4 — STANDARD

Data Retention Policy — Define & Publish

📐 D3 + D5 ⏱ 1 hr ⚡ Policy doc + privacy.html section
›
Problem

No data retention policy is defined. COPPA requires that personal information collected from children be retained only as long as reasonably necessary for the purpose it was collected, then deleted.

Solution

Define and publish a simple retention schedule:

  • Account data: Retained while account is active + 90 days after deletion request
  • Progress/quiz data: Retained while account is active; deleted with account
  • Analytics data: Firebase default (14 months); consider reducing to 2 months
  • Consent records: Retained for 3 years per FTC guidance

Add a "Data Retention" section to privacy.html. No system changes required — this is a policy publication.

Expected Score Impact
D3: +2 pts D5: +1–2 pts
